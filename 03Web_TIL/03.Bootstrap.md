
# Bootstrap TIL

## 0) Bootstrap을 왜 쓰나 

CSS를 직접 작성하면 원하는 디자인을 자유롭게 만들 수 있지만, 다음 문제가 반복된다. 

* 버튼/카드/폼/네비게이션 같은 “자주 쓰는 UI”를 매번 처음부터 만들기엔 시간이 너무 든다.
* 브라우저마다 기본 스타일이 달라서, 같은 HTML인데도 미묘하게 다르게 보이는 경우가 많다.
* 반응형(모바일/태블릿/PC)을 안정적으로 맞추려면 레이아웃과 규칙을 잘 잡아야 한다.

Bootstrap은 이 문제를 “툴킷(toolkit)”으로 해결한다.
즉,

> 미리 디자인된 CSS 규칙 + 재사용 가능한 컴포넌트 + 유틸리티 클래스들을 제공해서
> 최소한의 코드로 빠르게 UI를 만든다. 

---

## 1) Bootstrap이 정확히 뭐야?

Bootstrap은 **CSS 프론트엔드 프레임워크(툴킷)** 이다. 

핵심 특징을 한 문장으로 말하면:

* “이미 만들어진 디자인 시스템을 가져다 쓰고”
* “클래스 이름만 붙이면”
* “일관된 UI를 빠르게 만들 수 있다.”

여기서 포인트는 “클래스 이름을 쓰는 방식”이다.
Bootstrap은 CSS를 직접 길게 쓰는 대신, 미리 정의된 클래스(유틸리티/컴포넌트)를 조합해서 화면을 만든다. 

---

## 2) Bootstrap을 적용하는 방법 (CDN vs Download)

자료는 두 가지 사용법을 보여준다. 

### 2-1) CDN 방식 (가장 빠른 시작)

CDN(Content Delivery Network)은 전 세계에 분산된 서버에서 파일을 빠르게 내려주는 방식이다. 

Bootstrap을 CDN으로 쓰면, HTML `<head>`에 CSS와 JS를 “링크로 끼워 넣는 것만으로” 바로 사용 가능하다. 

감각적으로는 이렇게 이해하면 된다:

* “Bootstrap에서 만들어 둔 CSS 파일을 인터넷에서 불러와서”
* 내 HTML에서 그 CSS 클래스를 갖다 쓰는 구조

장점

* 설정이 거의 필요 없고 바로 시작 가능
* 수업/실습/간단 프로젝트에 특히 편함 

단점

* 네트워크에 의존
* 커스터마이징(빌드 단계에서 변수 수정 등)에는 한계가 있음

---

### 2-2) Download 방식 (파일을 프로젝트에 직접 포함)

Bootstrap 파일(또는 dist)을 프로젝트에 내려받아 포함시켜 사용하는 방식이다. 

장점

* 오프라인/내부망에서도 안정적
* 커스터마이징/빌드 파이프라인과 결합하기 좋음

단점

* 초기 세팅이 CDN보다 번거로움

실전에서는 “프로젝트 규모/배포 방식/커스터마이징 필요성”에 따라 선택한다.

## 3) Bootstrap이 “Reset CSS”를 포함하는 이유

브라우저는 각자 기본 CSS(user agent stylesheet)를 가지고 있다. 그래서 같은 HTML이라도 브라우저별로 기본 margin/padding/폰트가 달라질 수 있다. 

Bootstrap은 이런 차이를 줄이기 위해 “reset/normalize” 성격의 스타일을 포함한다. 자료에서는 Bootstrap의 reset이 `bootstrap-reboot.css`로 소개된다. 

정리하면:

* Reset/Normalize를 통해 브라우저 기본 스타일의 차이를 줄이고
* 그 위에 Bootstrap의 컴포넌트/유틸리티 스타일이 안정적으로 적용되게 만든다. 

---

## 4) Bootstrap의 핵심 사용 방식: “유틸리티 클래스”

Bootstrap을 쓰면 CSS를 길게 쓰는 대신, 클래스 이름으로 스타일을 붙인다.
자료는 특히 **Spacing(여백)** 유틸리티를 강조한다. 

### 4-1) Spacing 유틸리티가 왜 중요하나?

UI를 “그럴듯하게” 만드는 1순위는 사실 색보다도 **간격(여백)** 이다.
간격만 제대로 잡아도 화면이 훨씬 정돈돼 보인다.

Bootstrap은 margin/padding을 클래스 조합으로 빠르게 준다.

### 4-2) Spacing 클래스 구조(공식)

자료 표를 말로 풀면 이런 규칙이다. 

* `m` = margin, `p` = padding
* 방향(sides):

  * `t` top, `b` bottom, `s` start(왼쪽), `e` end(오른쪽), `x` 좌우, `y` 상하, (없으면 전체)
* 크기(size):

  * `0 ~ 5` (작을수록 좁고, 클수록 넓음)
  * 예: `mt-5`, `px-3`, `my-2`

예시로 감 잡기:

* `mt-5` : 위쪽 margin을 크게
* `p-3` : 전체 padding을 중간 정도
* `px-4` : 좌우 padding을 크게
* `my-1` : 위아래 margin을 아주 조금

이 방식에 익숙해지면 “CSS 파일 열기 전에” 대부분의 간격 문제를 HTML 클래스만으로 해결할 수 있다.

---

## 5) Typography: 글자(제목/본문/리스트)를 빠르게 정돈

자료는 Typography 파트에서 “기본 텍스트 요소에 대한 스타일”과 “display headings”를 보여준다. 

핵심 메시지는 이거다:

* HTML의 기본 heading(h1~h6)만으로도 의미 구조를 잡되,
* Bootstrap의 타이포그래피 클래스로 “보기 좋은 크기/굵기/간격”을 빠르게 맞출 수 있다. 

특히 display heading은 “훨씬 큰 타이틀”이 필요할 때 쓰는 스타일 패턴이다. (랜딩 페이지 메인 문구 같은 것)

---

## 6) Colors: 색상 시스템이 ‘정해져 있다는 것’의 장점

Bootstrap은 색을 자유롭게 아무거나 쓰는 게 아니라, **일관된 컬러 팔레트(의미 기반)** 를 제공한다. 

대표적인 의미 기반 색상:

* primary(주요 동작), secondary(보조), success(성공), danger(위험/삭제), warning(주의), info(정보), light/dark 등 

이 시스템의 장점:

* 버튼, 배지, 알림 등 여러 UI에서 색 의미가 일관돼서 사용자 경험이 좋아짐
* 개발자는 “예쁜 색”보다 “의미 있는 색”으로 선택하면 됨

Bootstrap은 텍스트/배경/테두리 색상을 각각 클래스 형태로 제공한다. (예: text-*, bg-* 같은 패턴) 

---

## 7) Components: “자주 쓰는 UI가 이미 완성돼 있다”

자료는 Bootstrap 컴포넌트를 “Bootstrap에서 제공하는 UI 요소 모음”으로 설명한다. 

예시로 카드(Card), 버튼(Button), 배지(Badge), 네비게이션 바(Navbar) 등이 언급된다. 

여기서 중요한 사고방식:

* 컴포넌트는 단순히 “예쁘다”가 아니라
* **재사용 가능한 UI 패턴**을 이미 검증된 형태로 제공한다는 의미다.

즉, 디자인/마크업을 처음부터 만들지 않아도 “형태가 이미 정리된 UI”를 가져다 쓰고, 필요하면 커스터마이징하면 된다.

---

## 8) Semantic Web & Semantic HTML: “div로만 만들지 말고 의미를 담자”

자료는 Bootstrap 이야기에서 갑자기 **Semantic Web / Semantic HTML**로 확장한다. 이게 중요한 이유는, UI 프레임워크를 쓰더라도 “HTML의 의미”는 계속 중요하기 때문이다. 

### 8-1) Semantic Web

Semantic Web은 “웹에 존재하는 정보를 의미(semantic) 중심으로 표현해서, 기계(검색엔진/도구)가 더 잘 이해하도록 하는 방향”으로 설명한다. 

### 8-2) Semantic HTML이란?

> 태그 자체가 의미를 가진 요소를 사용해서 문서 구조를 표현하는 것 

대표적인 semantic 요소:

* `header`, `nav`, `main`, `section`, `article`, `aside`, `footer` 

왜 중요하냐면:

* 접근성(스크린리더)에서 “이 영역은 네비게이션이다 / 본문이다”를 구분 가능
* 검색엔진이 페이지 구조를 이해하기 쉬움
* 팀원이 코드를 읽을 때도 “의도가 드러나는 구조”가 된다

결론:
Bootstrap을 쓴다고 해서 div만 잔뜩 쓰는 게 아니라, **의미 있는 구조 + Bootstrap 클래스** 조합이 가장 좋다.

---

## 9) Semantic in CSS: CSS도 ‘의미 단위’로 설계해야 한다

자료는 “CSS 방법론”으로 넘어가며 semantic in CSS를 이야기한다. 

핵심 감각은:

* CSS 클래스도 “그냥 모양을 설명하는 이름”만 쓰면 재사용이 어렵고 유지보수도 힘들다.
* 의미/역할 중심으로 재사용 가능하게 쪼개는 사고가 필요하다.

이 맥락에서 OOCSS가 등장한다.

---

## 10) OOCSS(Object Oriented CSS): Bootstrap 클래스 설계가 왜 이런가?

자료는 OOCSS를 “객체 지향적으로 CSS를 작성하는 방법”으로 소개한다. 

여기서 가장 중요한 2가지 원칙이 나온다. 

### 원칙 1) 구조(Structure)와 스킨(Skin)을 분리

* 구조: 크기, 배치, 여백 같은 “틀”
* 스킨: 색, 그림자, 테두리 같은 “꾸밈”

이렇게 분리하면 좋은 점:

* 같은 구조에 다른 스킨을 쉽게 조합 가능
* 예: 같은 버튼 구조에 primary/success/danger 색상만 바꿔 재사용

### 원칙 2) 컨테이너(container)와 콘텐츠(content)를 분리

* 컨테이너는 “어떤 영역에 들어가든” 재사용 가능한 형태로 만들고
* 콘텐츠는 그 안에 들어가는 요소들

이 방식이 Bootstrap 유틸리티/컴포넌트 철학과 연결된다.
즉, Bootstrap이 클래스 조합으로 UI를 만드는 이유는 “조합 가능한 재사용 단위”를 만들기 위해서다.

## 11) Bootstrap을 쓸 때 생기는 습관(실전 팁)

자료에서 마지막에 “CSS를 처음부터 다 짜는 게 너무 복잡하면 Bootstrap을 활용하라”는 흐름이 나온다. 

실전에서 유용한 습관을 정리하면:

1. 레이아웃/간격부터 잡는다

   * `container`, `row`, `col` 같은 그리드든, 간단히 spacing 유틸리티든
   * “여백이 잘 맞으면 절반은 완성”이다.

2. 컴포넌트를 먼저 가져다 쓰고, 나중에 커스터마이징한다

   * 버튼/카드/폼은 직접 만들기 전에 Bootstrap 컴포넌트로 빠르게 구성

3. HTML은 semantic하게 쓰고, 디자인은 Bootstrap 클래스로

   * `header/nav/main/footer` 구조로 의미를 살리고
   * 그 위에 Bootstrap 클래스 붙이기

4. !important 남발 대신, 클래스 조합과 구조 설계를 먼저 본다

   * Bootstrap을 쓸 때 충돌이 나면 “명시도 싸움”으로 가기 전에
     클래스/구조를 재정리하는 게 더 건강한 해결이다.

## 12) 오늘 TIL 한 장 요약

* **Bootstrap**: 미리 정의된 CSS/컴포넌트/유틸리티로 UI를 빠르게 만드는 프레임워크(툴킷) 
* **CDN**: 외부 서버에서 CSS/JS를 링크로 불러와 빠르게 사용 
* **Reset/Normalize**: 브라우저 기본 스타일 차이를 줄여 일관된 UI 기반 만들기 (`bootstrap-reboot`) 
* **Utility class**: `mt-5`, `px-3` 같은 클래스 조합으로 spacing/색/정렬을 빠르게 적용 
* **Typography/Colors**: 의미 기반(성공/위험 등) 디자인 시스템 제공 
* **Components**: Button, Card, Badge, Navbar 같은 재사용 UI 요소 제공 
* **Semantic HTML**: `header/nav/main/section/article/footer`처럼 태그 의미로 구조 표현(접근성/SEO/가독성) 
* **OOCSS**: 구조/스킨 분리, 컨테이너/콘텐츠 분리 → 클래스 조합으로 재사용성 극대화
